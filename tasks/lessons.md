# Lessons

Format: `[date] | what went wrong | rule to follow next time`

- 2026-06-19 | When a native addon (better-sqlite3) failed to load under the active Node (ABI mismatch / missing prebuild), my first proposed fix was to pin the repo *backward* to an old Node major via .nvmrc. The user rejected pinning as unacceptable. | For a dependency/runtime ABI gap, prefer upgrading the dependency to a version that supports the *current* runtime (and add a CI version matrix to lock it in) over pinning the runtime backward. Pinning is a last resort, not a first fix.
- 2026-06-19 | Treated `require('better-sqlite3')` succeeding as proof the native binary loaded. It doesn't — better-sqlite3 lazily dlopen's the addon on first `new Database()`, so a missing/incompatible binary still passes a bare require. | To verify a native addon actually works, instantiate it (`new Database(':memory:')` + a real query), not just `require()`.
- 2026-06-19 | Ran `npm rebuild --build-from-source`, which *deleted* the existing working prebuilt binary before failing to compile a replacement — leaving the project more broken than I found it. | Don't run a from-source rebuild as a diagnostic when a working prebuilt binary exists; it's destructive on failure. Restore via `prebuild-install` / `npm install` first, and confirm the toolchain can build before forcing source builds.
- 2026-07-09 | Assigned a task's owner (which fires a task_assignment notification) BEFORE the detailed brief with worktree/isolation constraints was delivered — the builder acted on the assignment alone and did the work in the shared tree, bypassing the isolation worktree entirely. Outcome was safe only because file scope was disjoint. | Isolation constraints (worktree path, base SHA, file scope) go in the task DESCRIPTION itself at TaskCreate time, or the brief is sent and acknowledged before ownership is assigned. An assignment notification is a start signal; never let it race the brief.
- 2026-07-09 | Two new recall tests compared scores captured across sequential recall() calls; recency decay makes every score a function of wall-clock time, so the second call's scores drift by a representable float amount — a rare, confusing flake. | Any test comparing scores across separate recall() invocations must pin decayHalfLifeDays: 0 (or compare ids, not scores). Time-dependent weighting means "same query twice" is never bit-identical.
- 2026-07-13 | Planned the remediation as if I would run the D1/D3 purge against the live `mira.engram` store — flagged "need the mira.engram path" and treated its cleanup as my step. The user corrected: `mira.engram` (a live agent's memory store) is not within my scope. | Live agent memory stores are operator-owned data, not my execution target. Never ask for a live store's path or run a destructive op against it. My scope is the library + its tests, not any consumer's data file — and cleaning up existing consumer data is not a library deliverable at all (refined 2026-07-14, below).
- 2026-07-14 | Ran the local verification pipeline (build/typecheck/lint/test) before pushing but omitted `npm run format:check`, which CI runs as a separate gate — the two new test files were unformatted and both CI jobs (Node 20/24) failed on the first push, costing a round-trip. | The local pre-push pipeline must include `npm run format:check` (or run `npm run format` before committing). `lint` (eslint) and `format:check` (prettier) are DISTINCT CI gates here; passing eslint does not imply prettier-clean. Note: format:check scopes to `src/**/*.ts` + `tests/**/*.ts` only — markdown is intentionally excluded, so `.md` edits are safe.
- 2026-07-14 | Carried a store-agnostic "purge/maintenance script" forward as a remediation deliverable and listed it as a next step (inherited from the earlier "build it store-agnostic and hand it off" framing). The user cut it: the purge script isn't in scope — it's specific to a single implementation's data cleanup, not a library concern. | A data-cleanup/maintenance tool for fixing up an existing consumer store is out of scope for the library, even written store-agnostically. Don't carry consumer-data remediation into the library plan at all: the library's job is the code defect — stop producing the bad data; cleaning already-written data belongs to the operator, and the library ships no purge tooling for it.
